## Когда использовать
Применять при написании/рефакторинге любого Java-кода для плагинов Minecraft (Paper/Bukkit): GUI, конфиги, слушатели событий, работа со строками/цветом, командами, хранение данных в памяти.

## Общие принципы

- Целимся строго в **Paper API**, а не в голый Bukkit/Spigot API, если не указано иное — Paper даёт доступ к более быстрым и современным методам (Adventure из коробки, Registry API, Folia-scheduler, Data Components для айтемов).
- Не используем deprecated методы Bukkit, если у Paper есть современная замена (например, `Registry` API вместо `Material.valueOf`/статичных enum-геттеров там, где это применимо, современный Paper `ItemStack` Data Component API вместо ручного `ItemMeta` для новых версий 1.20.5+).
- Java 21+: используем records для неизменяемых DTO/конфиг-моделей, sealed classes/pattern matching где это уместно вместо цепочек instanceof.

## GUI

- Не используем голый `Inventory`/`InventoryClickEvent` вручную. Берём готовую Paper-native библиотеку GUI:
    - Приоритет: **InvUI** (`xyz.xenondevs.invui`) — написана специально под Paper, полноценно поддерживает Adventure Component, современные версии, разные типы инвентарей (chest, anvil, furnace и т.д.), не требует шейдинга легаси-конвертаций строк.
    - Альтернатива, если нужен более простой синтаксис под чисто chest-меню: **triumph-gui**.
    - Не пишем свою обвязку над `InventoryClickEvent`/`InventoryCloseEvent` с нуля, если задачу закрывает библиотека — это лишний источник багов (дюп-клики, десинхронизация состояния).

## Библиотеки и зависимости

- Все внешние библиотеки, которые публикуются в Maven Central / доступном репозитории, подключаем через `libraries:` в `plugin.yml` (Paper library loader скачивает их при старте), а не шейдим:
  ```yaml
  libraries:
    - xyz.xenondevs.invui:invui:VERSION
    - it.unimi.dsi:fastutil:VERSION
  ```
- Шейдим (relocate) только то, чего нет в публичных репозиториях, либо библиотеки, которые сами требуют релокейта из-за конфликтов classpath.
- **fastutil** — обязателен там, где есть коллекции с примитивными ключами/значениями (int→Object, UUID реже, но int slot-инвентаря, int id, long timestamp и т.д.). Используем `Int2ObjectOpenHashMap`, `Object2IntOpenHashMap`, `IntArrayList` и т.д. вместо `HashMap<Integer,...>`/`ArrayList<Integer>` — не боксим примитивы без необходимости.
- fastutil-коллекции **не потокобезопасны** — если объект пишется/читается с разных потоков (например, из async-задачи и главного потока), используем `java.util.concurrent` (`ConcurrentHashMap`, `CopyOnWriteArrayList`) либо явную синхронизацию, а не fastutil напрямую.

## События вместо тикающих менеджеров

- Никаких `runTaskTimer`/`scheduleAtFixedRate`, которые каждые N тиков проверяют состояние (например, "не истёк ли эффект", "не покинул ли игрок зону") — это лишняя постоянная нагрузка.
- Вместо polling — Bukkit/Paper Listener на конкретное событие: `PlayerMoveEvent` (с проверкой смены блока, не координаты!) для зон, `EntityDamageEvent`/`EntityRegainHealthEvent` для хп-завязанной логики, `PlayerQuitEvent`/`PlayerJoinEvent` для очистки/восстановления состояния.
- Если реально нужен отложенный одноразовый эффект (например, кулдаун скилла закончится через 10 сек) — используем **одну** отложенную задачу конкретно под это событие (`runTaskLater`), не постоянный цикл проверки.
- На Folia-совместимых сборках Paper — используем region-aware scheduler API (`GlobalRegionScheduler`, `EntityScheduler`, `RegionScheduler`), а не голый `BukkitScheduler`, если планируется поддержка Folia.

## Конфиг

- Конфиг читается **только** при `onEnable()` и при выполнении команды reload — больше никогда во время обычной работы плагина.
- Все считанные значения сразу мапятся в POJO/record и кэшируются в памяти (финальные поля или volatile-поле на весь конфиг целиком, если reload может произойти из другого потока).
- При reload — собираем новый объект конфига и атомарно подменяем ссылку (`volatile ConfigData config = newConfig;`), не мутируем поля старого объекта — это избавляет от гонок между чтением конфига в игровом потоке и его обновлением командой.

## Цвета/Adventure

- Красим строки один раз через Adventure, поддерживая все три формата: `&`-коды, `&#rrggbb` hex и MiniMessage-теги `<tag>`. Держим один общий helper:
  ```java
  private static final LegacyComponentSerializer LEGACY = LegacyComponentSerializer.builder()
      .character('&')
      .hexColors()
      .build();
  private static final MiniMessage MINI = MiniMessage.miniMessage(); // singleton, не создавать заново каждый вызов

  public static Component colorize(String raw) {
      return raw.contains("<") ? MINI.deserialize(raw) : LEGACY.deserialize(raw);
  }
  ```
- Красим строки **при парсинге конфига**, сразу сохраняем в кэш уже готовый `Component` (или `Component[]` для лора и т.п.) — не парсим цвет каждый раз при отправке сообщения игроку.
- Строки с PlaceholderAPI — красим в рантайме **после** подстановки плейсхолдеров, т.к. значение плейсхолдера может само содержать цветовые коды. Что можно оптимизировать: если плейсхолдеров в строке несколько, но раскрашиваемая обёртка одна — собираем финальную строку через `StringBuilder`/`String.format` один раз и парсим её в Component один раз, а не красим каждый кусок отдельно.
- Отправляем игрокам `Component`, а не `String` — `player.sendMessage(Component)`, `ItemMeta#displayName(Component)`, `ItemMeta#lore(List<Component>)` — избегаем обратной конвертации в legacy-строки там, где это не нужно (иначе смысл Adventure теряется).

## Команды

- Не пишем ручной `onCommand`/`TabCompleter` с цепочками `if (args[0].equals(...))` для нетривиальных команд с подкомандами и аргументами. Используем аннотационный command-фреймворк с Brigadier-интеграцией (например **Lamp** или **cloud**) — даёт автокомплит, парсинг аргументов, права на подкоманды без ручного бойлерплейта, и не тратит время на рефлексию в рантайме (парсинг аннотаций происходит один раз при регистрации).

## Хранение данных на игрока

- Данные игрока (баланс, выбранный кит, кулдауны скиллов и т.п.) держим в кэше в памяти (`Map<UUID, PlayerData>`, с UUID, не с именем игрока), синхронизируем с БД/файлом асинхронно, не блокируя главный поток при каждом чтении.
- `PersistentDataContainer` читаем один раз при входе игрока (`PlayerJoinEvent`) и кэшируем значение в объекте `PlayerData`, а не дёргаем PDC на каждый чих (PDC — это NBT под капотом, чтение недорогое, но не бесплатное на горячем пути).

## Итемы

- Для сборки `ItemStack` с NBT/PDC-данными и на новых версиях (1.20.5+) — используем Paper Data Component API (`ItemStack#setData`) там, где это покрывает потребность, вместо ручной работы с `ItemMeta` + `PersistentDataContainer`, когда данные представимы стандартным компонентом.
- Кэшируем шаблонные `ItemStack` (иконки меню, дефолтные киты) как готовые объекты при старте/reload, а не пересобираем `ItemStack` заново при каждом открытии меню игроком — клонируем (`ItemStack#clone()`) готовый шаблон.

## Чего избегаем

- `Bukkit.getOnlinePlayers()` можно свободно вызывать в горячих участках — это дешёвый view без пересборки коллекции.
  Не стоит только сохранять результат как долгоживущую ссылку между тиками или использовать из другого потока —
  итерация гарантированно безопасна только сразу же, в игровом потоке.
- Строковые сравнения `Material.valueOf(name)`/`EntityType.valueOf(name)` на горячем пути — резолвим один раз при загрузке конфига, храним готовый enum/объект.
- Синхронный файловый I/O (сохранение статистики, логов) в игровом потоке за пределами старта/reload — уводим в async-таск.
